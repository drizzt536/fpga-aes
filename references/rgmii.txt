terms:
 - RGMII: reduced gigabit medium-independent interface
 - TXCK: transmit clock
 - TXEN: transmit enable
 - RXCK: receive clock
 - RXDV: receive data valid
 - MDIO: management data input/output
 - MDC: management data clock
 - TXD: transmit data
 - RXD: receive data

The board PHY is an RTL8211F-CG. To reset, EPHY_RST_N has to be pulled low for 10ms, and then should not touch MDIO for the next 72ms (as of rev 1.7 of the Realtek docs).
The safest option would be to pulse reset when the FPGA fabric starts up, but it isn't required, so it is probably fine to jut drive it high combinationally and ignore it.

MDC is the clock for MDIO. driving it low combinationally means no MDIO transactions ever happen, so it will just be idle on the PHY. MDIO is for checking link status and link speed and stuff. MDIO can be left floating without issues.

For sending nibble `x`, TXD[i] should be `(x >> i) & 1`? So essentially, to send a nibble, it should just be `TXD <= nibble;`

diagram of TXD writes vs TXCK transitions:

 clk
  ^ www t WWW T www t WWW T www t WWW T www t
1 |     +-----+     +-----+     +-----+     +--
  |     |     |     |     |     |     |     |
  |     |     |     |     |     |     |     |
  |     |     |     |     |     |     |     |
0 | ----+     +-----+     +-----+     +-----+
  |---------------------------------------------> t (ns)
  |     4     8     12    16    20    24    28

The lowercase 't' are when the PHY reads TXD on the clock rising edge, and the uppercase 'T' are when the PHY reads TXD on the clock falling edge. the capitalization of 'w' corresponds to which adjacent transfer it is for.

Each w is 'www' instead of just 'w' since the TXD writes don't have to be exactly a 90 degree offset; they just have to be written at some point in the range of 4ns before the next transition.

TXCK is 125MHz for gigabit ethernet (which the PHY should always be running at).
After writing to TXD, the pins need time to settle, which is why the data writes have a +90 degree offset with respect to the clock edge.
The transfers are DDR.

 TXCK
  ^     : 1   2   3     126 127 128
1 |---+ : +---+   +-...-+   +---+ : +---
  |   | : |   |   |     |   |   | : |
0 |   +---+   +---+     +---+   +---+
  |--------------------------------------> t (ns)
  |   0 : 4   8   12    504 508 512 516
        :                         :
 TXEN   :                         :
  ^     :                         :
1 |     +-------------------------+
  |     |                         |
0 |-----+                         +-----
  |--------------------------------------> t (ns)
  |     ^                         ^
   transfer start            transfer end

TXEN has the same offset as TXD.
Once TXEN is lowered back to 0, it signals that the previous transfer ha the final nibble in the packet.
raise TXEN 2ns before TXCK transfer edge 1, and lower it 2ns after TXCK transfer edge 128.
This assumes there are 64 bytes in the packet, which is the minimum (6*2 + 2 + 46 + 4 bytes).

The whole point of the offsets between TXCK, TXEN, and TXD are so the PHY can just read everything on the clock edge, and all the values will be settled. For RXD, the PHY will do the same thing, so don't worry about any offset stuff on the reading side.
